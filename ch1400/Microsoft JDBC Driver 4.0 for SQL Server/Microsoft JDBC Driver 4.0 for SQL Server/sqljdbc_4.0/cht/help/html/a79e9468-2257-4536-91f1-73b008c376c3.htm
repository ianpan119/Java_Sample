存取擴充事件記錄檔中的診斷資訊

在 Microsoft JDBC Driver 4.0 for SQL Server 中，追蹤 (追蹤驅動程式作業) 已經更新，讓您能夠更輕易地將用戶端事件與伺服器連接性信號緩衝區中的診斷資訊 (例如連接失敗) 以及擴充事件記錄檔中的應用程式效能資訊相互關聯。如需有關讀取擴充事件記錄檔的詳細資訊，請參閱＜檢視事件工作階段資料＞。

詳細資料

Microsoft JDBC Driver for SQL Server 會針對連接作業傳送用戶端連接識別碼。如果連接失敗，您可以存取連接性信號緩衝區 (透過連接性信號緩衝區疑難排解 SQL Server 2008 中的連接性問題)、尋找 ClientConnectionID 欄位，並且取得有關連接失敗的診斷資訊。只有在發生錯誤時，信號緩衝區才會記錄用戶端連接識別碼 (如果連接在傳送預先登入封包之前就失敗，系統將不會產生用戶端連接識別碼)。用戶端連接識別碼是 16 位元組 GUID。如果 client_connection_id 動作已加入至擴充事件工作階段中的事件，您也可以在擴充事件目標輸出中尋找用戶端連接識別碼。如果您需要進一步的用戶端驅動程式診斷協助，可以啟用追蹤、重新執行連接命令，並且觀察追蹤中的 ClientConnectionID 欄位。

您可以使用 ISQLServerConnection 介面，以程式設計方式取得用戶端連接識別碼。連接識別碼也會出現在任何與連接有關的例外狀況中。

發生連接錯誤時，伺服器 BID 追蹤資訊和連接性信號緩衝區中的用戶端連接識別碼可協助您將用戶端連接與伺服器連接相互關聯。如需有關伺服器 BID 追蹤的詳細資訊，請參閱＜資料存取追蹤＞。請注意，這篇資料存取追蹤文章也包含執行資料存取追蹤的相關資訊，但是這些資訊不適用於 Microsoft JDBC Driver for SQL Server。如需有關使用 Microsoft JDBC Driver for SQL Server 進行資料存取追蹤的詳細資訊，請參閱＜追蹤驅動程式作業＞。

JDBC 驅動程式也會傳送執行緒特有的活動識別碼。如果擴充事件工作階段是在啟用 TRACK_CAUSAILITY 選項的情況下啟動，就會在工作階段中擷取活動識別碼。如果使用中的連接發生效能問題，您可以從用戶端的追蹤中取得活動識別碼 (ActivityID 欄位)，然後在擴充事件輸出中找出活動識別碼。擴充事件中的活動識別碼是附加了四位元組序號的 16 位元組 GUID (與用戶端連接識別碼的 GUID 不同)。此序號代表要求在執行緒中的順序。系統會針對 SQL 批次陳述式和 RPC 要求傳送 ActivityId。若要將 ActivityId 傳送至伺服器，您必須先在 Logging.Properties 檔案中指定下列機碼/值組：

com.microsoft.sqlserver.jdbc.traceactivity = on

如需詳細資訊，請參閱＜追蹤驅動程式作業＞。這個追蹤旗標會搭配對應的 JDBC 物件記錄器使用，以便決定是否要在 JDBC 驅動程式中追蹤和傳送 ActivityId。除了更新 Logging.Properties 檔案以外，您也必須將記錄器 com.microsoft.sqlserver.jdbc 啟用成 FINER 或更高設定。如果您想要針對特定類別所提出的要求，將 ActivityId 傳送至伺服器，就必須將對應的類別記錄器啟用成 FINER 或 FINEST。例如，如果類別是 SQLServerStatement，請啟用記錄器 com.microsoft.sqlserver.jdbc.SQLServerStatement。

下列範例會使用 Transact-SQL 來啟動擴充事件工作階段，這個工作階段將儲存在信號緩衝區中，並且針對 RPC 和批次作業記錄用戶端所傳送的活動識別碼：

create event session MySession on server
add event connectivity_ring_buffer_recorded,
add event sql_statement_starting (action (client_connection_id)),
add event sql_statement_completed (action (client_connection_id)),
add event rpc_starting (action (client_connection_id)),
add event rpc_completed (action (client_connection_id))
add target ring_buffer with (track_causality=on)
